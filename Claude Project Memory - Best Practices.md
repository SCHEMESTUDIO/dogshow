# Claude Project Memory & Recall — Best Practices

A reusable guide for setting up persistent context across multiple chats in a Claude project. Based on lessons learned building dogshow.lol.

---

## The Problem

Each new chat in a Claude project starts fresh. Without explicit setup, Claude will:
- Not know what files matter or what the project does
- Potentially edit the wrong files (e.g., old prototypes instead of the live codebase)
- Repeat mistakes from previous chats
- Ask you the same context questions every time

## The Solution: Two-Layer Architecture

### Layer 1 — CLAUDE.md (lives in your project folder)

This file sits in your project's root directory. Claude reads it automatically when working with your project files. It's the **codebase reference** — technical, factual, scannable.

**What goes in CLAUDE.md:**
- Warnings (e.g., "do NOT edit this file" — anything that prevents repeat mistakes)
- File map (what each file does, approximate line counts)
- Tech stack (languages, frameworks, hosting, services)
- API endpoints
- Key functions and their purpose
- Architecture decisions that aren't obvious from the code

**What does NOT go in CLAUDE.md:**
- Business context, goals, or strategy
- User preferences or working style
- Pending work or task status
- External references (Slack channels, dashboards, etc.)

**Template:**

```markdown
# [Project Name] — Codebase Reference

## ⚠️ CRITICAL WARNINGS
[Put your biggest footgun here. What file should never be edited?
What assumption will a fresh chat make that's wrong?]

---

## Tech Stack
- **Frontend**: [framework, language, build step?]
- **Backend**: [server, hosting, database]
- **Payments/Auth/etc.**: [key integrations]

## File Map

| File | Purpose |
|------|---------|
| `file.ext` | What it does |

## API Endpoints

| Method | Path | Purpose |
|--------|------|---------|
| GET | `/endpoint` | What it returns |

## Key Functions
- `functionName()` — What it does
```

**Key principle:** Only put things here that could drift from reality. If the code is self-explanatory, don't document it. Focus on what's *not* obvious.

### Layer 2 — Memory Files (persist across all chats)

Memory files live in Claude's internal memory directory. They load automatically at the start of every conversation. These store **context you can't derive from the code**.

**What goes in memory:**
- User preferences and working style
- Business context (goals, conversion targets, current stage)
- Pending work (specialist feedback, planned features, known bugs)
- Warnings and lessons learned (mistakes to never repeat)
- External references (where things live outside the repo — dashboards, Slack channels, docs)
- Pointers back to CLAUDE.md for codebase details

**What does NOT go in memory:**
- File structure, tech stack, code patterns (these belong in CLAUDE.md)
- Git history or recent changes (use `git log`)
- Anything documented in CLAUDE.md (don't duplicate)

**Memory file format:**

Each memory file uses frontmatter:

```markdown
---
name: Short descriptive name
description: One-line summary — used to judge relevance in future chats
type: [user | feedback | project | reference]
---

[Content here]
```

**Memory types:**
- `user` — Who you are, how you work, what you prefer
- `feedback` — Corrections and confirmed approaches (include **Why:** and **How to apply:**)
- `project` — Business context, pending work, decisions, current state
- `reference` — Where to find things in external systems

**MEMORY.md index:**

An index file that lists all memory files. Each entry should be one line, under 150 characters:

```markdown
- [Title](filename.md) — one-line hook describing what's in it
```

Keep this concise — it's loaded into every conversation. Lines after 200 get truncated.

## Recommended Memory Files for Any Project

Start with these five. Add more as the project grows.

### 1. User Profile (`user_profile.md`)
```markdown
---
name: User Profile
description: Role, preferences, working style — tailor responses accordingly
type: user
---

- [Your name], [your role/company]
- Prefers [brevity / detail / etc.]
- [Any specific preferences for how Claude should work with you]
```

### 2. Business Context (`project_context.md`)
```markdown
---
name: Project Business Context
description: What the project is, who it's for, current goals — can't be derived from code
type: project
---

## What Is It
[One paragraph: what does this project do?]

## Current Stage
[Launch? Growth? Rebuilding? What's the priority right now?]

## Key Goals
[What are you optimizing for? Conversions? Speed? Reliability?]
```

### 3. Pending Work (`pending_work.md`)
```markdown
---
name: Pending Work
description: Planned changes, feedback to apply, features in progress — NOT YET DONE
type: project
---

[List of planned changes with status (not started / in progress / done)]
```

### 4. Lessons Learned (`lessons_learned.md`)
```markdown
---
name: Lessons Learned
description: Mistakes and warnings from previous chats — prevent repeat errors
type: feedback
---

[What went wrong, why, and how to avoid it]

**Why:** [Root cause]
**How to apply:** [Specific rule for future chats]
```

### 5. External References (`references.md`)
```markdown
---
name: External References
description: Where things live outside the repo — hosting, services, dashboards, contacts
type: reference
---

- **Hosting**: [where and how]
- **Payments**: [provider, dashboard URL]
- **Analytics**: [tool, ID]
- **Contact**: [email, team channel]
```

## Layer 3 — Daily Auto-Updater (Scheduled Task)

Set up a recurring scheduled task that scans the codebase and keeps both layers in sync. This prevents drift between what the docs say and what the code actually does.

**When to create it:**
After your initial CLAUDE.md and memory files are in place.

**What the task should do:**
1. Read CLAUDE.md and all memory files
2. Scan project files for structural changes (new files, renamed files, removed files)
3. Check `git log` for recent changes
4. Update CLAUDE.md if the file map or tech stack changed
5. Update memory files if pending work status changed
6. Keep updates minimal — only change what's actually different

**Schedule:** Daily at a quiet time (e.g., 9am). The task runs automatically.

**Important:** Run the task manually once after creation to pre-approve its tool permissions. Otherwise it may pause on permission prompts during automatic runs.

## Anti-Confusion Patterns

These are specific patterns that prevent Claude from making costly mistakes in new chats.

### 1. Lead CLAUDE.md with Warnings
The very first section of CLAUDE.md should be the biggest footgun in your project. Examples:
- "DO NOT edit OldPrototype.jsx — the live site is static HTML"
- "The /legacy directory is deprecated — all new work goes in /src"
- "Never run migrations without checking with the team first"

### 2. Double-Layer Critical Warnings
For the most important warnings, put them in BOTH places:
- CLAUDE.md (read when touching project files)
- A `feedback` memory file (loaded at conversation start)

This way Claude hits the warning twice before it can do damage.

### 3. Pointer Memory Files
When you move codebase docs into CLAUDE.md, leave a pointer in memory:
```markdown
---
name: Codebase Docs Location
description: Points to CLAUDE.md for file structure and tech stack details
type: reference
---
All codebase documentation lives in CLAUDE.md at the project root. Read that file for technical questions.
```

This prevents future chats from wondering where the docs went.

### 4. Status Markers on Pending Work
Always mark pending work with clear status so new chats don't re-do or skip work:
- `NOT YET APPLIED`
- `IN PROGRESS — [what's done so far]`
- `COMPLETED — [date]`

## Setup Checklist for a New Project

1. [ ] Create the project folder in Claude and select it
2. [ ] Write `CLAUDE.md` in the project root (warnings first, then file map, tech stack, APIs)
3. [ ] Create memory files: user profile, business context, pending work, lessons learned, references
4. [ ] Create `MEMORY.md` index pointing to all memory files
5. [ ] Set up a daily scheduled task to keep docs in sync
6. [ ] Run the scheduled task once manually to pre-approve tool permissions
7. [ ] Test by starting a fresh chat and asking Claude about the project — verify it knows what's what

---

## Maintenance Rules

- **CLAUDE.md** is the source of truth for *what exists in the code*
- **Memory** is the source of truth for *why things are the way they are*
- Don't duplicate between the two layers
- When something changes in code, update CLAUDE.md (or let the daily task catch it)
- When a decision is made or feedback is given, save it to memory immediately
- Prune stale memories — old pending work that's been completed, outdated warnings
- Keep MEMORY.md index under 200 lines
