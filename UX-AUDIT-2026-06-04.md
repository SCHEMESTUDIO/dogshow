# The Dog Show — UX / Flow Audit

**Date:** 2026-06-04 · **Method:** Live walkthrough of production (dogshow.lol) via browser + real emails inspected in Gmail, cross-referenced against the repo. **Scope:** every public URL and every email a user receives during/after signup.

> A PartyKit redeploy was performed mid-audit (to rule out stale-server drift). Findings below note pre- vs post-deploy state where relevant. The static frontend is on Vercel; the realtime/API/email server is on PartyKit and is deployed separately.

---

## Summary

Tested fully (live): all public pages, free signup, magic-link login, welcome + login emails, unsubscribe endpoint, **a real $3.99 BYD purchase end-to-end (payment → provisioning → upload → certificate), and the scheduled-slot RSVP + reminder flow.**
Not live-tested: bones top-up ($1.99), out-of-bones, 2nd-dog 409 (code-traced only).

**Severity tally:** 1 High open (+1 High found-and-fixed) · 2 Medium · ~9 Low/Minor.

The single most important theme: **the deployed PartyKit server can drift from your repo.** The magic-link email failure was caused by this and was fixed by redeploying — but the redeploy did **not** fix the unsubscribe link, which means that one is a genuine current-code/bundle issue, not staleness.

**The BYD happy path works** end-to-end: real Stripe payment → `verify-checkout` provisioning → "You're in" email → username step → dog upload → AI classifier correctly tagged "Chocolate Lab" (0.93) → certificate email → pre-show page → RSVP → 5-min reminder email. The issues below are around the edges of that path, not the spine of it.

## HIGH

### H1 — Unsubscribe link is broken in every email (CAN-SPAM)
**Confidence: High (observed live, twice, pre- and post-redeploy).**
The unsubscribe URL delivered in emails is malformed — it is missing the `=` between `t` and the token, e.g.:
`.../unsubscribe?u=user_kiruu1&tD961cefab83b77e`  (should be `&t=<token>`).
Hitting it returns *"This unsubscribe link is missing data."* Reconstructing `&t=<visible token>` returns *"invalid or has been tampered with"* — so the token itself is also wrong/truncated, not just the `=`.

- Affects **every** user-facing email (all share `unsubscribeFooter()`).
- The repo source (`party/server.js:3410`, `:3417`) builds a correct `&t=${token}` link, and `git status` is clean — yet production emits the broken form, **and a full redeploy did not fix it** (while it *did* fix the magic-link issue, proving the deploy applied). So this is not staleness.
- **Action:** investigate the deployed bundle specifically — add a temporary log of the exact `partyUnsubUrl` string the server generates, and confirm `_unsubToken()` returns a 16-hex value. Re-test from a fresh email. This is a legal/compliance item, so treat as priority.

### H2 — Magic-link login email not delivered  → **FIXED during audit**
**Confidence: High.** Before the redeploy, a registered account requested a login link and none arrived (the welcome email to the same inbox arrived in seconds). After redeploy, the same pre-existing account received its link and logged in successfully end-to-end. Root cause was server drift (stale build not sending / not indexing). **Verify it stays fixed**, and confirm no pre-deploy real customers are stuck (spot-check that an older paid account can request a link and receive it).

## MEDIUM

### M1 — US visitors are getting EU cookie-consent treatment (lost ad data) — ROOT CAUSE FOUND
**Confidence: High (root cause identified in git).** On a real US visit the EU banner showed and GA/Bing UET were gated **off** (`window.gtag` undefined). Root cause: the **committed/deployed** middleware is `middleware.mjs`, but **Vercel only runs middleware named `.ts/.tsx/.js/.jsx`** — a `.mjs` is silently served as a static file and never executes. So the `dogshow_geo` cookie is never set, and `consent.js` hits its EU fail-safe for everyone. The fix (`middleware.ts` + `tsconfig` include) was made locally but **never committed** — it's sitting untracked in the working tree.
- **Fix:** `git rm middleware.mjs`, commit the untracked `middleware.ts` + the `tsconfig.json`/`CLAUDE.md`/`index.html` changes, push → Vercel runs the real middleware → `dogshow_geo` gets set → US visitors get ROW (no banner, trackers on).
- **Impact:** until shipped, GA4 + Bing UET conversions for US visitors aren't captured unless they click Accept — undercounts your ad-trial data.
- This is the same repo↔prod drift theme as H1/H2.

### M2 — success.html swallows payment-verification failures ✅ FIXED
**Confidence: High (code + live).** `success.html` defaulted `tier` to `'general'` when params were missing and silently redirected to `show.html?topup=1`; a failed `verify-checkout` was only `console.error`'d. Fixed: `registerUser()` now returns success/failure, and a `showVerifyError()` state appears on any failure (or missing `session_id`) — a calm, accurate message ("If you were charged, your access activates within a few minutes — reply to your Stripe receipt and we'll sort it out") instead of a silent redirect. Bones-pack buyers are now only redirected to the show on actual success.
- Still open (separate): `app.js` has no `?topup=1` handler, so the intended "+250 bones" confirmation toast doesn't fire. Minor; left for later.

### M3 — BYD dog upload is gated behind the "Pick a name" username modal (abandonment = paid-but-no-dog) ✅ FIXED (deployed; full live re-test pending)
**Confidence: High (reproduced live in the real purchase).** After paying, the dog is **not** uploaded on the success page. It uploads only after the buyer enters the show **and** clears the mandatory "Pick a name" username modal — at which point `/upload-dog` fires (confirmed 200). In the window before that, `/my-dog` returns `dog: null` and the stashed photo is already cleared from `localStorage`. So if a premium buyer closes the tab at the username modal (or it errors), **their dog is never created and won't retry on next visit** — they're a paid account with no dog. This is almost certainly a source of the "stuck paid users" your admin tooling exists to resolve.
- Compounding it: `success.html` already shows **"✓ Entered into The Dog Show"** *before* the dog actually exists — a premature confirmation.
- **Action:** persist the pending dog until `/upload-dog` actually succeeds (don't clear on read), and/or submit the dog immediately on `verify-checkout` rather than gating it behind the username step. At minimum, don't claim "entered" until the server confirms it.

## LOW / MINOR

### L1 — `login.html` dead-ends on a stale/expired link ✅ FIXED
Visiting `/login` with no/expired token showed *"Invalid login link…"* with no way to recover. Fixed: all three error states now reveal a **"Request a new link →"** button that deep-links to the landing-page sign-in form (`/#loginSection`).

### L2 — Magic-link email footer is invisible (white-on-white) ✅ FIXED
The login email had a light background but reused the dark-email footer (white text), making the unsubscribe link + address unreadable. Fixed: the login email is now dark-themed (`#1a1035`) like the welcome/certificate emails, so the footer is legible.

### L3 — Login email copy says "expires in 15 minutes" but the token is valid 60 minutes ✅ FIXED
Copy corrected to "60 minutes" to match the token TTL.

### L4 — At least one aired dog has `slug: null` ("Skeeterino")
Its gallery card works (links via `dog.html?id=…`), but it has no working SSR `/d/{slug}` certificate and no per-dog OG share image. Looks like an early upload before the slug/username fixes. Backfill a slug, or confirm such dogs can't be shared in a broken state.

### L5 — Gallery links to client-rendered `dog.html?id=…`, not the canonical SSR `/d/{slug}`
Both render, but the gallery bypasses the share-optimized SSR cert pages. Consider pointing gallery cards at `/d/{slug}` (falling back to `?id=` only for null-slug dogs) for consistent share previews + SEO.

### L6 — ~5s blank "Waiting for the show to start…" on first show load
On first connect the stage is black until the first WebSocket `sync` arrives (~5s), then populates normally. Minor first-impression risk for cold paid traffic — consider showing a seeded/last-known dog or a nicer loading state.

### L7 — Cookie banner can intercept the hero "Enter Your Dog" CTA
On first visit the upload modal would not open until the cookie banner was dismissed. Worth confirming the banner isn't blocking first-click conversions on the primary CTA.

### L8 — Certificate email is wrong for *scheduled* dogs ✅ FIXED
A dog booked for a future slot got the certificate email **immediately** on upload, with copy that said *"now in the rotation, collecting bones"* and a **"View certificate"** link — but the dog hadn't aired, and that link leads to the **pre-show countdown** page. Fixed in `sendCertificateEmail` (`party/server.js`): scheduled dogs now get scheduled-aware copy — subject "🗓️ … is booked", eyebrow "Booked", "booked for {UTC time}", a "View … page" CTA, and an "RSVP and cheer them on" share line. Immediate ("show now") uploads are unchanged. The upload handler now passes `slotAt`/`firstAppearedAt` to the email so the branch can fire. **Needs a PartyKit deploy.**

### L9 — Duplicate RSVP reminder email
The "🎬 Louise is on in 5 minutes" reminder was delivered **twice** (two distinct message IDs, same second). Looks spammy; dedupe the reminder send (guard per `{dogId,userId,sent5m}`).

### L10 — RSVP promises a "1 hour before" reminder that can't happen for near-term slots
The RSVP confirmation and reminder copy always say *"an hour and 5 minutes before."* For a dog booked < 60 min out (the picker allowed ~6 min out), the 1-hour reminder is impossible. Make the promise conditional on how far out the slot is.

### L11 — RSVP silently sends a second "Welcome" email
RSVP auto-registers the email as a free user, which fires a full Welcome email on top of the RSVP confirmation + reminder. Three emails from one RSVP may feel like a lot; consider suppressing the welcome when the registration originates from an RSVP.

### L12 — RSVP overwrites the current session token ✅ FIXED
`api/dog.js` set `dogshow_token` to the RSVP-issued (new free) account unconditionally, so a **logged-in owner who RSVPs to any dog gets silently swapped off their own account**. Fixed: the RSVP session is now only adopted for a logged-*out* visitor (`!localStorage.getItem('dogshow_token')`); an existing session is never overwritten.
